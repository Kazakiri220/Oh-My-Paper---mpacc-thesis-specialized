# Case Analysis Framework

Choose frameworks because they fit evidence and the research question, not because they are fashionable.

## Case Design First

Name the case design before choosing a framework.

Single-case designs:

- Extreme/unusual: the case deviates from common practice, rules, or routine events.
- Illuminating: the case reveals a new phenomenon or hard-to-observe situation.
- Longitudinal: the case can be divided into stages and compared across time.

Multi-case designs:

- Competitive: similar background, different behavior paths, differentiated outcomes.
- Equifinal: different starting points or paths, similar outcomes.
- Two-tail: opposite extreme cases.
- Variation: controls irrelevant differences and focuses on the key variable.

Honesty rules for multi-case work:

- State ownership, customer strategy, business model, financing channel, life-cycle stage, and other material differences.
- If more than three structural differences exist, do not call the design "variation".
- If both cases improve, do not call the design "two-tail".
- If outcomes point in the same direction, do not call the design "equifinal" unless the analytical goal is truly same-result explanation.
- Treat life-cycle differences as factors or boundary conditions, not as facts to hide.
- Do not overclaim strict causal identification. Use comparison and process tracing to build a plausible mechanism.

## Framework Index

| Framework | Fits | Avoid When |
|---|---|---|
| COSO / internal control five elements | internal control, project control, audit control | no process or control evidence |
| Full-process/life-cycle analysis | cost, project, budget, carbon accounting | the case has no stage-based data |
| Balanced scorecard | performance evaluation beyond finance | no customer/process/learning data |
| DuPont/EVA/financial ratios | profitability, debt, performance effect | thesis claims are mainly process/governance |
| Event study | market reaction to litigation, disclosure, reform | no clear event date or market data |
| fsQCA | multiple condition configurations | too few comparable cases or no calibrated variables |
| Social network analysis | related-party identification, hidden relationships | relationship data cannot be collected |
| Machine learning / text analysis | warning, classification, prediction | model output cannot be interpreted for accounting decisions |
| Business model canvas | data assets, platform businesses | the accounting issue is unrelated to value creation path |
| Resource orchestration | digital transformation path and capability formation | only static performance comparison is needed |

## Depth Principles

1. Quantify where possible: ratios, deviations, frequencies, weights, warning scores, event windows, or calibrated conditions.
2. Compare horizontally or vertically: peer firms, industry averages, pre/post event, policy stages, or project life-cycle stages.
3. Explain accounting meaning: connect data to recognition, measurement, disclosure, control, audit evidence, cost behavior, risk response, or performance.

Good: "For an intelligent cost warning thesis, use data warehouse design to collect production-cost drivers, then compare model warnings with actual abnormal cost periods."

Weak: "Describe the company's history and then say digital tools improve management."

## Framework Output

For every framework, state:

- input data;
- analytical steps;
- output indicators or findings;
- how outputs feed problem, cause, or solution chapters.

## Full-Read Pattern Additions

- Optimization cases need old-state evidence, diagnosed problem, cause, design, application or expected effect, and safeguard.
- Mechanism/effect cases need path variables, effect indicators, and a boundary between direct effect and spillover.
- Accounting-treatment cases need economic substance, standards mapping, recognition timing, measurement attribute, presentation/disclosure, and case effect.
- Intelligent model cases need input data, indicator construction, model choice, validation/application, interpretability, and management action.
- Staged transformation cases need driver, resource, capability, and effect for every stage.

Good: "For a resource-orchestration thesis, divide the case into stages and show how each stage converts resources into capabilities and measurable value."

Weak: "Tell the company's transformation history and then list several financial ratios."
