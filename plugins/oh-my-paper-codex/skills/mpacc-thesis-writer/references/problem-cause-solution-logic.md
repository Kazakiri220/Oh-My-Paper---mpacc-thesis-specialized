# Problem Cause Solution Logic

Use a strict alignment table. Do not let recommendations float away from evidence.

## Clarify The Research Problem First

Do not start from a broad topic or a list of management issues. Before using the alignment table, write:

- reality problem: the observable gap, anomaly, deterioration, decision conflict, or process failure;
- key associated factor(s): one main factor, two at most;
- research question: `observable phenomenon + associated factor(s) + consequence`;
- provisional answer/main claim: a statement about why the problem occurs or how it can be solved.

The provisional answer must not simply reverse the problem wording. "Low quality -> improve quality" is not a research claim. "Single historical-average forecasting causes procurement budget deviation because business drivers are not linked to project demand" is a research claim.

## Alignment Table

| Diagnosed problem | Evidence | Cause | Solution | Safeguard | Expected effect |
|---|---|---|---|---|---|
| What is wrong | data/process/interview/comparison | why it happens | what to change | people/system/process/policy | how to verify |

Every row must be filled before drafting the recommendation chapter.

## Problem Rules

- Problems are evidence-visible symptoms: high deviation, missed disclosure, delayed response, weak indicator coverage, model misclassification, unclear recognition boundary, negative FCF, rising debt pressure, regulatory inquiry, or control failure.
- Problems must be directly visible in standard indicators or public/case materials. Do not invent a concept or reconstruct an indicator only to make the problem appear.
- Causes explain mechanisms beyond identities or common sense: maturity mismatch, budget-driver disconnection, data silo, incentive mismatch, insufficient expertise, unclear responsibility, weak system integration, incomplete standards.
- Solutions are interventions: process redesign, indicator library, accounting treatment model, risk scoring, disclosure template, governance arrangement, technical platform.
- Keep the main thesis focused. A dissertation can mention secondary issues, but the research problem should revolve around one main relationship or mechanism.

Good: "Risk identification misses non-financial information because the bank only uses financial statements and guarantee records; add tax, judicial, transaction, and text-data interfaces with data quality controls."

Weak: "Strengthen awareness, improve management, and increase investment."

## Recommendation Test

For each recommendation, ask:

- Which problem does it solve?
- Which cause does it address?
- What evidence shows it is needed?
- What material, department, system, or indicator makes it implementable?
- How will success be measured?

Remove recommendations that fail this test.

For tool applications, add:

- Which industry-specific indicator or threshold is used?
- What is the basis for the threshold?
- Who operates the tool?
- What data path feeds it?
- What trigger action follows?

For decision problems, compare feasible alternatives and explain why the final option is chosen. A single slogan-like recommendation is not a solved problem.
